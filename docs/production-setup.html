What lives where

  • Supabase — hosts the PostgreSQL database and all auth. No separate DB to provision.
  • AWS App Runner — runs the Fastify API inside a Docker container pulled from ECR.
  • ECR (Elastic Container Registry) — stores the API Docker images.
  • Expo EAS — builds and submits the iOS/Android app binaries.
  • Anthropic — called by the API for AI mnemonic generation.

Supabase — Database & Auth

①

Create a new Supabase project

Go to supabase.com → New Project. Choose a region close to your App Runner region (e.g. us-east-1 → choose AWS East US).

②

Collect credentials

From Project Settings → API:

  • Project URL — used in the mobile app as EXPO_PUBLIC_SUPABASE_URL
  • anon public key — used in the mobile app as EXPO_PUBLIC_SUPABASE_ANON_KEY
  • service_role key — used in the API as SUPABASE_SERVICE_ROLE_KEY (keep secret)
  • JWT Secret — from Settings → API → JWT Settings → SUPABASE_JWT_SECRET

From Project Settings → Database → Connection string → URI:

  • Use the pooler connection string (port 6543) for production → DATABASE_URL
③

Run Drizzle migrations

This creates all tables from the schema.

cd /path/to/kanji-learn
DATABASE_URL="postgresql://..." pnpm --filter @kanji-learn/db db:migrate
④

Enable email auth

In Supabase dashboard → Authentication → Providers → Email → ensure enabled. Optionally disable email confirmations for easier onboarding initially.

API Server — AWS App Runner

ℹ️ All commands below run from the monorepo root (kanji-learn/). The Docker build context must be the root because the Dockerfile copies from packages/.
①

Set the App Runner service ARN

Open scripts/deploy-api.sh and fill in the APPRUNNER_SERVICE_ARN variable after you create the service in step ④. Leave it blank for the first run.

②

First push — create the ECR repo and push the image

The deploy script creates the ECR repo automatically if it doesn't exist.

export AWS_REGION=us-east-1
./scripts/deploy-api.sh

This will log in to ECR, create the kanji-learn-api repo, build the image for linux/amd64, and push it. The App Runner trigger is skipped on first run since the service doesn't exist yet.

③

Create the App Runner service (AWS Console)

  • Go to AWS Console → App Runner → Create service
  • Source: Container registry → Amazon ECR
  • Image URI: <account>.dkr.ecr.us-east-1.amazonaws.com/kanji-learn-api:latest
  • ECR access role: create new (App Runner needs pull access)
  • Deployment trigger: Manual (the deploy script handles this)
  • Port: 3000
  • Instance: 0.25 vCPU / 0.5 GB to start (~$15/mo)
  • Health check path: /health
  • Add all environment variables from Section 5
④

Copy the service ARN and update the deploy script

After the service is created, copy its ARN from the console (format: arn:aws:apprunner:us-east-1:123456789:service/kanji-learn-api/abc123) and paste it into scripts/deploy-api.sh as APPRUNNER_SERVICE_ARN.

⑤

Verify the service is healthy

# App Runner provides a URL like: https://abc123.us-east-1.awsapprunner.com
curl https://<your-apprunner-url>.awsapprunner.com/health
# Expected: {"ok":true}
⑥

Custom domain (optional)

In App Runner console → your service → Custom domains → associate your domain. App Runner provisions and auto-renews the TLS certificate.

Environment Variables

API Server — set in App Runner console under "Environment variables"

VariableDescription

postgresql://postgres.[ref]:[password]@aws-0-us-east-1.pooler.supabase.com:6543/postgres
SUPABASE_JWT_SECRET REQUIRED From Supabase → Settings → API → JWT Secret. Used to verify auth tokens.
SUPABASE_SERVICE_ROLE_KEY REQUIRED From Supabase → Settings → API. Used for admin operations. Keep secret.
ANTHROPIC_API_KEY REQUIRED From console.anthropic.com. Used for AI mnemonic generation.
EXPO_ACCESS_TOKEN REQUIRED From expo.dev → Account Settings → Access Tokens. Used to send push notifications.
CORS_ORIGIN OPTIONAL Comma-separated allowed origins. Defaults to *. Set to your App Runner URL for production.
INTERNAL_SECRET REQUIRED Shared secret for the POST /internal/daily-reminders endpoint. Must match the value configured in the EventBridge Lambda function. Generate with: openssl rand -hex 32
LOG_LEVEL OPTIONAL Pino log level. Default: info. Use warn in production to reduce noise.
NODE_ENV SET BY DOCKER Already set to production in the Dockerfile.

Mobile App — set in apps/mobile/.env.local (dev) or EAS Secrets (prod)

VariableDescription
EXPO_PUBLIC_API_URL REQUIRED Your App Runner URL, e.g. https://abc.us-east-1.awsapprunner.com
EXPO_PUBLIC_SUPABASE_URL REQUIRED From Supabase → Settings → API → Project URL
EXPO_PUBLIC_SUPABASE_ANON_KEY REQUIRED From Supabase → Settings → API → anon public key
⚠️ EXPO_PUBLIC_* variables are baked into the app binary at build time. To update them you must rebuild and resubmit the app.

Database Migration

⚠️ Run migrations before the API server starts for the first time, and before each deploy that includes schema changes.
①

Run Drizzle migrations (creates all tables)

DATABASE_URL="postgresql://..." pnpm --filter @kanji-learn/db db:migrate
②

Run the social features migration (one-time)

Adds the friendships table and email column to user_profiles. Safe to re-run — all statements use IF NOT EXISTS.

DATABASE_URL="postgresql://..." npx tsx apps/api/migrate_social.ts
# Expected output: Social migration applied

After confirming success this script can be deleted.

③

Seed kanji data (one-time)

Required — the app has no content without it.

DATABASE_URL="postgresql://..." pnpm --filter @kanji-learn/db seed

Ongoing Deployments

Deploy a new API version

After setting APPRUNNER_SERVICE_ARN in the deploy script, every future API deploy is a single command run from the monorepo root:

./scripts/deploy-api.sh

This rebuilds the Docker image, pushes to ECR, and triggers a zero-downtime rolling redeployment on App Runner. Typically completes in 2–4 minutes.

Deploy a new mobile app version

  • JS-only change — use an OTA update (no App Store review required):
cd apps/mobile
eas update --branch production --message "describe the change"
  • Native change (new plugin, new permission, dependency with native code) — requires a full EAS build and App Store resubmission.

Schema changes

Run migrations before deploying the API version that depends on them:

# 1. Apply migration to production DB
DATABASE_URL="postgresql://..." pnpm --filter @kanji-learn/db db:migrate

# 2. Then deploy the API
./scripts/deploy-api.sh

Daily Reminders — EventBridge Scheduler

Push notification reminders are no longer tied to the API process. A standalone Lambda function fires at 20:00 UTC daily via EventBridge Scheduler and calls the API's protected POST /internal/daily-reminders endpoint.

ℹ️ The Lambda contains no database credentials and no workspace packages — it's a single index.mjs file that makes one HTTP call. Deploying it takes under 5 minutes.
①

Generate a shared secret

openssl rand -hex 32

Save the output — you'll need it in two places: the App Runner env var INTERNAL_SECRET and the Lambda env var INTERNAL_SECRET.

②

Set INTERNAL_SECRET on App Runner

In the App Runner console → your service → Configuration → Environment variables, add:

INTERNAL_SECRET = <your-secret>

Then redeploy the API so the new env var takes effect:

./scripts/deploy-api.sh
③

Create the Lambda IAM role (first time only)

aws iam create-role \
  --role-name lambda-basic-execution \
  --assume-role-policy-document '{
    "Version":"2012-10-17",
    "Statement":[{"Effect":"Allow","Principal":{"Service":"lambda.amazonaws.com"},
    "Action":"sts:AssumeRole"}]
  }'

aws iam attach-role-policy \
  --role-name lambda-basic-execution \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
④

Package and deploy the Lambda

# From monorepo root
cd apps/lambda/daily-reminders
zip daily-reminders.zip index.mjs

aws lambda create-function \
  --function-name kanji-learn-daily-reminders \
  --runtime nodejs20.x \
  --handler index.handler \
  --zip-file fileb://daily-reminders.zip \
  --role arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):role/lambda-basic-execution \
  --timeout 60 \
  --environment 'Variables={
    API_BASE_URL=https://YOUR-APP-RUNNER-URL,
    INTERNAL_SECRET=YOUR-SECRET
  }'

For subsequent updates use update-function-code:

zip daily-reminders.zip index.mjs
aws lambda update-function-code \
  --function-name kanji-learn-daily-reminders \
  --zip-file fileb://daily-reminders.zip
⑤

Create the EventBridge Scheduler rule

# Get the Lambda ARN
LAMBDA_ARN=$(aws lambda get-function \
  --function-name kanji-learn-daily-reminders \
  --query 'Configuration.FunctionArn' --output text)

# Create the schedule (20:00 UTC daily)
aws scheduler create-schedule \
  --name kanji-learn-daily-reminders \
  --schedule-expression "cron(0 20 * * ? *)" \
  --flexible-time-window '{"Mode":"OFF"}' \
  --target "{
    \"Arn\": \"$LAMBDA_ARN\",
    \"RoleArn\": \"arn:aws:iam::$(aws sts get-caller-identity --query Account --output text):role/AmazonEventBridgeSchedulerRole\"
  }"
⚠️ EventBridge Scheduler needs its own IAM role to invoke Lambda. Create it in the console (Scheduler → Create schedule → IAM role → Create new) or use the AWS-managed role if available.
⑥

Test the end-to-end flow

# Invoke the Lambda manually to verify it reaches the API
aws lambda invoke \
  --function-name kanji-learn-daily-reminders \
  --log-type Tail \
  --query 'LogResult' \
  --output text \
  /dev/null | base64 -d

You should see [daily-reminders] Success in the tail output. Check App Runner logs to confirm the internal route was called.

Production Hardening Checklist

CORS origin lockdown

Set CORS_ORIGIN env var to your App Runner URL. Currently defaults to * which allows all origins.

Rate limiting — use Redis backend

Current in-memory rate limit resets on every container restart. Replace with Redis using @fastify/rate-limit's Redis store. Elasticache Serverless (~$0.008/GB-hr) is the easiest option.

DB connection pooling

Use the Supabase pooler connection string (port 6543, Transaction mode) as DATABASE_URL. Supabase's built-in PgBouncer handles connection limits for you.

Secrets management

Move ANTHROPIC_API_KEY, SUPABASE_SERVICE_ROLE_KEY, and DATABASE_URL to AWS SSM Parameter Store (SecureString) or AWS Secrets Manager. Reference them in App Runner rather than storing as plain env vars.

Email privacy for friend search

Any authenticated user can look up any email in user_profiles. Add an opt-out flag (allow_friend_search boolean) or hash emails before storing, and filter in the search query.

Leaderboard query performance at scale

The global top-10 query scans all of user_kanji_progress. Add a materialised view or a scheduled summary table refreshed nightly before user count grows large.

Streak timezone

computeStreak uses server UTC. Store a user timezone field and offset the day boundary so streaks don't break for users in UTC- timezones who study at night.

Rotate push notification credentials

Rotate EAS and APNs credentials before App Store submission. APNs keys expire and invalid keys cause silent push failures.

Wire Drizzle migrations into CI/CD

Currently migrations are run manually. Add a migration step to the deploy pipeline so schema changes are never accidentally deployed without the accompanying migration.

App Store privacy nutrition label

NSLocationWhenInUseUsageDescription is set. Ensure the App Store Connect privacy questionnaire reflects that location data is collected and stored (linked to user identity).

Troubleshooting

SymptomFix
Docker build fails: cannot find module '@kanji-learn/db' Build context must be monorepo root, not apps/api/. Run: docker build -f apps/api/Dockerfile .
App Runner shows unhealthy Check that all required env vars are set. App Runner logs → look for DATABASE_URL required or missing keys.
aws ecr get-login-password fails Run aws sts get-caller-identity to verify CLI credentials are valid.
Mobile app can't reach API Verify EXPO_PUBLIC_API_URL has no trailing slash and uses https://. Rebuild the app if the env var was changed.
Push notifications not arriving Verify EXPO_ACCESS_TOKEN is set on the API server. Check Expo push receipt logs.
Auth tokens rejected (401) Verify SUPABASE_JWT_SECRET matches the value in Supabase → Settings → API → JWT Secret exactly.
Social migration error: column already exists The script is idempotent — IF NOT EXISTS guards all statements. Re-run safely.
App Runner cold start slow Upgrade to 0.5 vCPU / 1 GB or enable "Always on" mode in App Runner health check settings.
Daily reminders Lambda returns 401 Verify INTERNAL_SECRET matches exactly in both the App Runner env vars and the Lambda env vars. Redeploy the API after changing App Runner env vars.
EventBridge schedule isn't triggering Check the schedule state in EventBridge → Schedules (must be Enabled). Verify the Scheduler IAM role has lambda:InvokeFunction permission on the function ARN.
